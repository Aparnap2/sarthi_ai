# ADR-001: Temporal for Workflow Orchestration (not Celery/APScheduler)

## Status
Accepted

## Date
April 2026

## Context
The guardian needed durable, retryable execution for agent workflows that could fail mid-flight — during LLM calls, DB writes, or Slack delivery. A worker crash mid-workflow must not lose state. The system processes investor metrics, detects anomalies, generates insights, and delivers them to founders; any step can fail due to network timeouts, rate limits, or infrastructure instability.

## Options Considered
| Option | Pros | Cons |
|--------|------|------|
| Celery + Redis | Familiar ecosystem, Python-native, well-documented, easy to reason about | No durable execution — a task that fails mid-LLM-call loses all context; retry logic must be hand-rolled |
| APScheduler | Simple cron-like scheduling, minimal infra | No workflow state, no retries, no recovery after crash — fire and forget |
| Temporal | Event-sourced durable workflows, automatic replay, built-in retries, workflow state survives worker crashes | Significant infra complexity (separate server, Go-based), steep learning curve, overkill for simple tasks |
| Prefect | Modern orchestration UI, Python-first, good developer experience | Less mature for long-running workflows, still requires external server, less battle-tested for agent patterns |

## Decision
Temporal. The key differentiator is its event-sourcing model: workflow state survives worker crashes. A Celery task that fails mid-LLM-call loses all context; Temporal replays from the last checkpoint, re-executing only the failed step with full state intact.

## Consequences
**Positive:**
- Workflow state is fully durable — a pod restart during an LLM call doesn't lose the investor being processed
- Built-in retry policies with exponential backoff handle transient LLM API failures
- Automatic replay means debugging is straightforward: replay the workflow with the same input and observe the failure
- Activity heartbeats provide visibility into long-running tasks (e.g., "fetching 30 days of metrics")
- Deterministic workflow code means the same input always produces the same execution path

**Negative / Tradeoffs:**
- Adds significant infrastructure complexity: a separate Temporal server and protobuf definitions
- For a portfolio with 10 beta users, this is overkill — the operational burden outweighs the reliability benefit at this scale
- Debugging replay issues requires understanding Temporal's deterministic replay semantics, which are non-obvious
- The Temporal server itself is implemented in Go, but workflows and activities are written in Python using the `temporalio` SDK (see apps/ai/src/worker.py)
- Local development requires running the Temporal dev server alongside PostgreSQL, Qdrant, and Redis

## What I'd Do Differently
For the first prototype, I'd start with APScheduler + PostgreSQL-persisted state. Add Temporal when the "what if the worker dies during an LLM call" problem becomes real and measurable. I added Temporal too early — it cost a week of infrastructure setup that didn't move the product needle for the first 10 users.
