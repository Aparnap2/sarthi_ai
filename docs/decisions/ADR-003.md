# ADR-003: DSPy for Prompt Management (not raw f-strings)

## Status
Accepted

## Date
April 2026

## Context
Guardian message quality needed to be consistent, testable, and improvable. Raw prompts are invisible — you can't unit-test a string template. The system generates investor-facing insights that must be accurate, empathetic, and actionable; prompt quality directly impacts product quality.

## Options Considered
| Option | Pros | Cons |
|--------|------|------|
| Raw f-string prompts | Zero dependencies, trivial to write, no learning curve | Invisible and untestable — a string template can't be validated, versioned, or systematically improved; quality regressions are silent |
| LangChain PromptTemplate | Structured prompt definitions, variable interpolation, some validation | Rigid abstraction tied to LangChain ecosystem; doesn't solve prompt optimization, just organization |
| DSPy Signatures | Declarative input/output fields with descriptions; optimizer finds the best prompt; versionable and comparable across experiments | Learning curve for the DSPy paradigm; overkill for simple summarization tasks; adds dependency weight |

## Decision
DSPy Signatures. The key differentiator: prompts are versionable and comparable. You define input/output fields with descriptions, and the optimizer finds the best prompt. A raw f-string is forever; a DSPy signature can be improved without rewriting the prompt.

## Consequences
**Positive:**
- Prompts are declarative: `GuardianInsight(input: Metrics, output: str)` makes the contract explicit — what goes in, what comes out
- Versionable and comparable: you can track which signature version produced better messages, and A/B test prompts systematically
- The optimizer can improve output quality across hundreds of LLM calls without manual prompt tweaking
- Output validation via Pydantic means malformed LLM responses are caught and retried, not silently passed to the founder
- Signatures can be composed: a `SummarizeMetrics` signature feeds into a `GenerateInsight` signature, each independently optimizable

**Negative / Tradeoffs:**
- DSPy adds a learning curve. For a simple "summarize these numbers" task, an f-string works fine and is immediately readable by any Python developer
- DSPy pays off when you need to systematically improve output quality across hundreds of LLM calls — at 10 beta users, the optimization machinery is underutilized
- Debugging a bad DSPy output requires understanding the optimizer's internal state, which is less transparent than reading a raw prompt string
- The DSPy dependency ties the prompt layer to a specific framework; migrating away means rewriting every signature

## What I'd Do Differently
I'd use DSPy for the GuardianInsight signature from day one (because message quality IS the product). For simpler agents like the QA agent's initial template matching, I'd start with an f-string and migrate to DSPy only when quality issues emerged.
