# ADR-005: Guardian Watchlist as Pure Python Rules (not LLM judgment)

## Status
Accepted

## Date
April 2026

## Context
Detecting "churn > 3%" is deterministic. Using an LLM to decide if churn is dangerous adds latency (2–5s), cost ($0.01–0.10 per check), and non-determinism (the LLM might change its mind) to a binary threshold check. The guardian runs 17 pattern checks on every pulse cycle — each one must be fast, free, and consistent.

## Options Considered
| Option | Pros | Cons |
|--------|------|------|
| LLM-evaluated patterns | Flexible — can detect fuzzy patterns like "MRR growth feels sluggish"; adapts to context without code changes | Slow (2–5s per check), costly ($0.01–0.10 per check), non-deterministic (the LLM might change its mind between runs), impossible to unit-test deterministically |
| Hybrid LLM + rules | Rules for known thresholds, LLM for edge cases; best of both worlds | Complex routing logic — deciding "does this need an LLM?" is itself a hard problem; doubles the code paths to test and maintain |
| Pure Python thresholds | Deterministic, instant (< 1ms), free, trivially unit-testable; 17 patterns run in under 1ms total | Can't detect fuzzy patterns like "MRR growth feels sluggish"; requires code changes to add or modify patterns |

## Decision
Pure Python. The GuardianDetector runs all 17 patterns in under 1ms with zero LLM calls. The LLM is reserved exclusively for generating the insight message — the trigger is purely rule-based.

## Consequences
**Positive:**
- All 17 seed-stage failure patterns evaluate in under 1ms total — zero latency impact on the pulse cycle
- Zero LLM cost for detection — the LLM is only called once per cycle for insight generation, not 17 times for threshold checks
- Fully deterministic: the same metrics always produce the same detection result; no "the LLM changed its mind" surprises
- Trivially unit-testable: each pattern is a pure function `(metrics: dict) -> bool` with predictable inputs and outputs
- Patterns are visible and auditable: a founder can see exactly which threshold triggered their alert (e.g., "churn was 4.2%, threshold is 3%")

**Negative / Tradeoffs:**
- Pure Python means you can't detect fuzzy patterns like "MRR growth feels sluggish." But the 17 seed-stage failure patterns are all threshold-based by definition — they're the known unknowns
- Unknown unknowns would need LLM judgment, but we don't have those patterns yet — and if we did, the rules engine would need redesign
- Adding a new pattern requires a code change and deploy, not a prompt tweak — slower iteration for pattern tuning
- The rules engine encodes assumptions about what "bad" looks like; if the definition of bad changes, the code must change

## What I'd Do Differently
Nothing. This is the right call and the one I'm most confident about.
