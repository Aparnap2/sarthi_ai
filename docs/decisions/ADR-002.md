# ADR-002: LangGraph for Agent State (not CrewAI/AutoGen)

## Status
Accepted

## Date
April 2026

## Context
Each guardian agent (Pulse, Anomaly, Investor, QA) has 5–7 nodes that pass typed state between them. Debugging requires seeing what each node produced. The system needed explicit, inspectable state transitions for multi-step agent reasoning — not opaque agent memory.

## Options Considered
| Option | Pros | Cons |
|--------|------|------|
| CrewAI | Role-based abstraction, high-level API, quick to prototype | Hides state inside agent memory that can't easily be inspected or checkpointed; hard to debug intermediate steps |
| AutoGen | Multi-agent conversation patterns, good for collaborative reasoning | Conversation-based state is unstructured; difficult to enforce deterministic workflows or add HITL gates |
| LangGraph StateGraph | Typed state dict that persists across nodes; checkpointable, debuggable, resumable; conditional edges and loops are first-class citizens | Requires thinking in graphs and state dicts; adds ceremony for simple flows |
| Raw function chains | Simple, transparent, 20 lines of code | No structure — conditional logic, retries, and state passing become spaghetti; impossible to checkpoint mid-chain |

## Decision
LangGraph StateGraph. The key differentiator is a typed state dict that persists across nodes — you can checkpoint, debug, and resume. CrewAI's role-based model hides state inside agent memory you can't easily inspect.

## Consequences
**Positive:**
- Every node's input and output is visible in the state dict — debugging a bad insight means tracing exactly which node produced the wrong data
- Conditional edges enable branching logic: "if anomaly detected, run AnomalyAgent, else skip" is a declarative edge, not an if-statement buried in code
- Checkpointing allows resuming workflows after failures — a crashed Pulse agent can restart from the last completed node
- HITL (human-in-the-loop) gates are native: interrupt the graph, wait for user input, resume
- State schema is Pydantic-typed, so invalid state transitions are caught at runtime

**Negative / Tradeoffs:**
- LangGraph requires you to think in graphs and state dicts. For a simple "fetch data → generate text → send message" flow, a function chain would be 20 lines vs 150 lines of LangGraph
- The graph abstraction pays off when agents get complex (conditional edges, loops, HITL gates) — but adds ceremony before complexity justifies it
- Learning curve: developers familiar with function chains must learn graph semantics (edges, conditions, state reducers)
- Testing individual nodes requires constructing partial state dicts, which is more setup than testing a standalone function

## What I'd Do Differently
I'd start with a simple function chain for the first agent. Add LangGraph when the agent needs conditional logic ("if anomaly detected, run AnomalyAgent, else skip"). I added LangGraph on day one — it added ceremony before complexity justified it.
