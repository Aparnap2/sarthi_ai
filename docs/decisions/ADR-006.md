# ADR-006: Slack-First Delivery (no dashboard)

## Status
Accepted

## Date
April 2026

## Context
Solo founders don't open monitoring dashboards. They live in Slack. Building a web dashboard adds weeks of frontend work for a surface that won't be checked, while Slack is the only async channel where the founder's attention already lives.

## Options Considered
| Option | Pros | Cons |
|--------|------|------|
| Web dashboard (React/HTMX) | Rich visualizations, historical trends, settings page, full control over UX | Weeks of frontend work for a surface solo founders won't check; requires auth, hosting, responsive design, and ongoing maintenance |
| Email digest | Universal reach, no platform dependency, works for non-Slack users | Low engagement — founders get hundreds of emails daily; a guardian alert buried in an inbox is a missed alert; no interactive feedback loop |
| Slack-first with Block Kit | Zero UI to build; rich message format (headers, sections, dividers, buttons); founder's attention is already there; interactive actions become the feedback loop | Slack-only means no historical view, no trend charts, no settings page; if the founder has no Slack, onboarding fails |
| Mobile push notifications | Highest immediacy, works offline, platform-native UX | Requires building/maintaining a mobile app or integrating with a push service (FCM/APNs); noisy — founders mute push notifications from non-critical apps |

## Decision
Slack Block Kit with interactive buttons. Zero UI to build. Rich message format (headers, sections, dividers, buttons). Actions (acknowledge/dismiss) become the feedback loop. If the founder has no Slack, onboarding fails loudly — no fallback.

## Consequences
**Positive:**
- Zero frontend UI to build — the guardian delivers insights through Slack's Block Kit API, which handles formatting, layout, and interactivity
- Interactive buttons (acknowledge/dismiss) create a built-in feedback loop: the system learns which alerts the founder cares about
- Slack is where founders already live — the guardian meets them in their existing workflow instead of asking them to learn a new tool
- Block Kit messages are structured and parseable — the feedback data (which buttons were clicked, when) feeds directly into the procedural memory layer
- Onboarding is a single step: connect Slack workspace. No dashboard setup, no email preferences, no mobile app download

**Negative / Tradeoffs:**
- Slack-only delivery means no historical view, no trend charts, no settings page — the founder can't "look back at last month's alerts" without scrolling Slack
- For a beta, this is the right constraint — it forces the guardian to be useful in the moment, not after the fact. But it limits the product's perceived completeness
- If the founder's team isn't on Slack, the product is unusable. The "fail loudly" onboarding is honest but means lost users
- Slack's API rate limits and message formatting constraints shape the guardian's output — the product is coupled to a single platform's capabilities
- No offline access: if Slack is down (rare but possible), the guardian has no delivery channel

## What I'd Do Differently
I'd build the Telegram fallback for development/testing (which I did). But for beta, I'd remove it entirely — if Slack isn't wired, the product shouldn't silently deliver to Telegram mock. It should fail the onboarding check.
