# ADR-004: 5-Layer Memory Spine Design

## Status
Accepted

## Date
April 2026

## Context
A guardian needs to distinguish between "what just happened" (working memory), "what happened before" (episodic), "what patterns connect" (semantic), "what the agent learned" (procedural), and "historical summaries" (compressed). A single Qdrant collection can't do this — each layer has different TTL, retrieval semantics, and write frequency.

## Options Considered
| Option | Pros | Cons |
|--------|------|------|
| Single Qdrant collection | Simple architecture, one dependency, easy to reason about | All memories share the same retrieval semantics — can't differentiate between "what happened today" and "what pattern emerged over 3 months" |
| Redis + Qdrant (2-layer) | Fast session memory (Redis) + persistent embeddings (Qdrant); covers most use cases | No causal reasoning (semantic), no learned patterns (procedural), no compression |
| Full 5-layer spine (Redis → Qdrant → Kuzu → PG → Qdrant compressed) | Each layer optimized for its purpose; enables causal reasoning, pattern learning, and efficient retrieval; designed for incremental infra adoption | 3 of 5 layers are stubs in production today; adds 3 dependencies (Redis, Kuzu, PostgreSQL) for memory that isn't yet utilized |

## Decision
5-layer spine with graceful fallback. Each layer has an `available()` check — if Redis isn't running, WorkingMemory falls back to an in-memory dict. If Kuzu isn't installed, SemanticMemory is a no-op. The architecture is designed for incremental infra adoption: the system works with zero memory layers, and each layer you add improves quality.

## Consequences
**Positive:**
- Graceful degradation: the system functions with no memory layers, and each added layer incrementally improves insight quality
- Each layer has optimal retrieval semantics: L1 (fast key-value), L2 (vector similarity), L3 (graph traversal), L4 (SQL aggregation), L5 (compressed vector search)
- TTL is per-layer: working memory expires in minutes, episodic persists for months, compressed summaries persist indefinitely
- The design is aspirational — the code structure exists before the infra, making it easy to add layers as needs emerge
- Clear separation of concerns: each memory layer has a distinct `write()`, `retrieve()`, and `clear()` interface

**Negative / Tradeoffs:**
- Most of these layers are empty in production today. The 5-layer design is aspirational — it makes the codebase look more complex than the running system
- The value is in the fallback pattern: the system works with zero memory layers, and each layer you add improves quality. But the fallback code itself adds complexity
- Kuzu (graph database) is a niche dependency — fewer developers know it, and its Python bindings are less mature than Neo4j or NetworkX
- Testing requires mocking 5 memory layers, which increases test setup complexity

## What I'd Do Differently
I'd start with just L2 (Qdrant episodic) and L5 (Qdrant compressed). Add L1 (Redis) when you need session persistence. Add L3 (Kuzu) when you need causal chain reasoning. Add L4 (PG procedural) when you need to optimize prompts per tenant. I built all 5 at once — 3 of them are stubs.
